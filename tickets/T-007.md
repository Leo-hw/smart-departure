## 목표

폴링 방식(매 시간 체크) 대신, 아침에 한 번 당일 일정을 분석해 알림 시각을 미리 계산하고 예약한다.
알림은 "준비 시작"과 "출발" 두 단계로 분리한다.
배포 환경은 Oracle crontab 또는 GitHub Actions 중 하나를 선택해 사용할 수 있도록 설계한다.

## Acceptance Criteria

### 스케줄링 로직
- [x] `core/scheduler.py` 구현 — 당일 일정 분석 후 알림 계획 목록 반환
- [x] 알림 종류: `준비 시작` / `출발`
- [x] 출발 알림 시각 = 일정 시작 - 이동 시간 - `buffer_minutes`
- [x] 준비 알림 시각 = 출발 알림 시각 - `prep_minutes` (settings.yaml 설정값, 0이면 준비 알림 생략)
- [x] 현재 시각이 각 알림 시각의 ±`alert_window_minutes` 이내이면 전송
- [x] 알림 계획을 `.runtime/schedule_today.json`에 저장 (당일 날짜가 다르면 재계산)

### 알림 메시지 포맷

- [x] 이동 시간이 추정값(`is_estimated=True`)이면 메시지에 "(이동 시간 추정)" 표기

- [x] 두 방식 모두 동일한 `main.py`를 진입점으로 사용 (배포 방식에 따른 코드 분기 없음)

## 구현 상세

### settings.yaml 추가 항목
```yaml
schedule:
  run_mode: "check"         # "check": 현재 시각 기준 알림 판단 (Oracle/GHA 공통)
  alert_window_minutes: 15  # 알림 시각 ±N분 이내이면 전송
  prep_minutes: 60          # 준비 시작 알림 (0이면 비활성)
  buffer_minutes: 10        # 도착 후 여유 시간
  dedup_ttl_minutes: 60
```

### 알림 계획 파일 (.runtime/schedule_today.json)
```json
{
  "date": "2026-04-17",
  "plans": [
    {
      "event_id": "abc123",
      "summary": "스터디 모임",
      "location": "강남구 역삼동 OO카페",
      "event_start": "2026-04-17T16:00:00+09:00",
      "travel_minutes": 35,
      "is_estimated": false,
      "departure_time": "2026-04-17T15:15:00+09:00",
      "prep_alert_time": "2026-04-17T14:15:00+09:00"
    }
  ]
}
```

### 실행 흐름
```
main.py 실행
  ↓
schedule_today.json 존재 + 오늘 날짜 일치? → 재사용
  아니면 → Calendar + Maps API로 재계산 후 저장
  ↓
현재 시각 기준으로 plans 순회
  각 plan의 prep_alert_time / departure_time이 ±alert_window_minutes 이내?
    → dedup 체크
    → 알림 전송
```

### GitHub Actions 워크플로우
```yaml
# .github/workflows/departure_check.yml
on:
  schedule:
    - cron: '0,30 * * * *'   # 30분 간격
  workflow_dispatch:           # 수동 실행 가능
```

### Oracle crontab 설정
```cron
*/30 * * * * cd /home/ubuntu/smart-departure && /home/ubuntu/smart-departure/.venv/bin/python main.py >> /var/log/smart-departure.log 2>&1
```
