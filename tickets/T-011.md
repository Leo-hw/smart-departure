# T-011 — [핫픽스] 당일 추가 일정 미반영 (스냅샷 staleness)

**상태**: 🟡 READY  
**담당**: Codex  
**의존**: T-010  
**우선순위**: 🔥 긴급 (모든 당일-추가 일정의 알림 누락)

---

## 배경 / 증상

2026-06-07 15:51에 오늘 17:00 코엑스 일정(장소 정상 입력)을 추가했으나, 이후 GitHub Actions 실행이 계속 `Today's plans: 0`을 출력하고 알림이 오지 않음.

캘린더 일정 자체는 정상 (location 채워져 있고 당일 범위 안). 원인은 코드.

## 근본 원인

`core/scheduler.py`의 `load_or_build_daily_schedule()`:
```python
if snapshot.get("date") == current_time.date().isoformat():
    return _deserialize_plans(snapshot.get("plans", []))   # 날짜만 보고 통째 재사용
```
- 당일 첫 실행에서 만든 스냅샷(그 시점 일정 기준, 흔히 빈 plans)을 **날짜가 같다는 이유만으로 하루 종일 재사용** → 캘린더를 다시 조회하지 않음
- T-010이 `.runtime`를 GHA `actions/cache`로 영속화하면서, 과거엔 휘발되어 매번 재생성되던 스냅샷이 이제 **하루 종일 살아남음** → staleness 표면화
- 결과: 당일 첫 실행 이후 추가된 일정은 그날 내내 안 보이고, 날짜가 바뀌면 이미 과거 → 영구 누락

## Acceptance Criteria

- [ ] 당일 중간에 추가된 일정이 **다음 실행(≤ cron 간격)에 plans에 반영**된다
- [ ] `load_or_build_daily_schedule()`가 매 실행 캘린더를 재조회하도록 변경 (날짜-only 재사용 제거)
- [ ] **dedup 상태(`sent_alerts.json`) 영속화는 그대로 유지** — T-010 동작 회귀 없음
- [ ] 이동시간은 기존 maps 캐시(6h TTL)로 재사용해 불필요한 Maps 호출 억제
- [ ] (선택) `schedule.snapshot_ttl_minutes` 설정 추가, 기본값 `0`(항상 재빌드). 향후 디스크 영속(Oracle) 환경에서 재사용 조절용
- [ ] 단위 테스트: "스냅샷 날짜=오늘이지만 새 일정 존재 → 재빌드되어 새 일정 포함" 케이스
- [ ] 단위 테스트: dedup 상태가 재빌드와 무관하게 유지되는지 회귀 테스트

## 구현 상세

### 핵심 변경
```python
def load_or_build_daily_schedule(settings, now=None):
    current_time = _ensure_kst(now or _current_time_kst())
    ttl = int(settings.get("schedule", {}).get("snapshot_ttl_minutes", 0))

    snapshot = _load_schedule_snapshot()
    if ttl > 0 and _snapshot_is_fresh(snapshot, current_time, ttl):
        return _deserialize_plans(snapshot.get("plans", []))

    # 기본(ttl=0): 항상 재빌드 → 당일 추가 일정 즉시 반영
    plans = build_daily_plans(settings, now=current_time)
    _save_schedule_snapshot(current_time.date(), plans)
    return plans
```
- `_snapshot_is_fresh`: 스냅샷에 `built_at`(ISO) 저장 후, `now - built_at <= ttl`일 때만 fresh
- 스냅샷은 계속 기록(관측/디버깅용)하되, **기본 동작은 매 실행 재빌드**

### 주의
- 스냅샷 재사용을 끄는 것이지 **스냅샷 파일/포맷을 없애는 게 아님** (catch-up·관측용 유지)
- dedup은 `core/dedup.py`의 `sent_alerts.json`이 담당 — 본 티켓에서 건드리지 말 것
