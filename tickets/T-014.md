# T-014 — 외부 트리거로 실행 신뢰성 확보 (cron-job.org → workflow_dispatch)

**상태**: 🟡 READY  
**담당**: Codex (검증) + 사용자 (PAT/cron-job.org 셋업)  
**의존**: T-012, T-013  
**우선순위**: 🔥 높음 (알림 정시성 핵심)

---

## 배경

GHA `schedule:` cron은 GitHub이 저우선순위로 취급해 실측 1.5~4.7시간 간격으로만 실행됨(요청은 `*/5`). `*/5`로 올려도(T-012) 효과 없음 — public 전환은 비용만 풀고 스케줄러 우선순위는 안 바꿈.

**핵심 사실**: 불안정한 건 `schedule:` 트리거뿐. `workflow_dispatch`(API/버튼)는 **즉시 실행**됨.
- 라이브 측정: dispatch 요청 → run 생성 **2초**, 즉시 in_progress.
- 반면 schedule run은 몇 시간 간격.

→ 신뢰성 있는 외부 시계(cron-job.org)가 매 5분 `workflow_dispatch`를 호출하면, 실행은 계속 GHA(무료)에서 하되 **정시성 확보**. 새 서버 불필요.

## 구현/설정 상세

### 코드 변경
거의 없음. `workflow_dispatch:`는 이미 워크플로우에 존재. concurrency 가드도 T-012에서 적용됨.
- (선택) `schedule:` cron을 `*/5` → `0 * * * *`(매시간) 백업으로 낮춰 무의미한 schedule 실행 노이즈 감소. 외부 트리거가 주(主)가 되므로.

### 셋업 절차
→ `deploy/external_trigger_setup.md` 참조 (PAT 발급 + cron-job.org 구성 + 검증).

### 보안 메모
- PAT는 fine-grained, **이 repo만**, **Actions: Read and write**만 (최소 권한).
- 유출 시 가능한 최악 = "내 워크플로우 실행/취소"뿐. 코드/Secrets 열람·push 불가 → 저위험.
- PAT는 cron-job.org에만 저장, 절대 커밋 금지. 만료 설정 + 주기적 로테이션 권장.

## 한계 / 후속
