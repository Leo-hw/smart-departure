# T-012 — GHA 실행 신뢰성 개선 (빈도 ↑ + catch-up 보강)

**상태**: 🟡 READY  
**담당**: Codex  
**의존**: T-010, T-011  
**우선순위**: 🔥 높음 (현재 알림이 실측상 안 옴)

---

## 배경

운영 중 GitHub Actions 스케줄 cron(`0,30 * * * *`, 30분 요청)이 실측 **~85분 간격**으로 띄엄띄엄 실행되고(요청의 ~65% 누락) 시각도 제멋대로라, 출발/준비 알림 윈도우를 통째로 건너뛰어 **알림이 오지 않음**.

repo를 **public으로 전환** 완료 → Actions 무제한 무료가 되어 고빈도 cron이 비용 부담 없이 가능해짐. 이를 활용해 신뢰성을 끌어올린다.

> 참고: 외부 트리거(cron-job.org 등)는 이번 범위 제외(보류). 순수 GHA 튜닝으로 먼저 개선하고, 부족하면 후속 티켓에서 외부 트리거 도입.

## Acceptance Criteria

### A. 실행 빈도 ↑ (public 무료 활용)
- [ ] `.github/workflows/departure_check.yml` cron `0,30 * * * *` → `*/5 * * * *` (5분 간격 요청)
- [ ] `workflow_dispatch` 수동 트리거 유지

### B. 실행 비용/속도 최적화
- [ ] `actions/setup-python`에 `cache: 'pip'` 적용 (의존성 설치 시간 단축)
- [ ] 동시 실행 방지: 워크플로우에 `concurrency` 그룹 추가 (느린 실행과 다음 tick 겹침 방지)
  ```yaml
  concurrency:
    group: departure-check
    cancel-in-progress: false
  ```

### C. catch-up 윈도우 보강 (틱 지연/누락 안전망)
- [ ] 출발 catch-up 만료를 설정값으로: `schedule.departure_catchup_minutes` (기본 45, 기존 고정 30 대체)
- [ ] 준비 catch-up은 기존대로 `departure_time`까지 유지
- [ ] `alert_window_minutes`는 15 유지
- [ ] 설정 변경이 `classify_alert`에 반영되고 단위 테스트로 검증

### D. 검증
- [ ] 머지 후 수 시간 동안 Actions 실행 타임스탬프가 실제로 ~5~15분 간격으로 들어오는지 확인
- [ ] 미래 일정 1건으로 준비/출발 알림이 실제 Discord로 도착하는지 확인

## 구현 상세

### 비용 메모
- public repo는 GitHub Actions **무제한 무료** → `*/5`(288회/일)도 비용 0
- 단, GHA 스케줄러는 여전히 best-effort라 5분 요청해도 실제론 더 띄엄띄엄일 수 있음. 그래서 C(윈도우 보강)로 안전망을 둠.

### classify_alert 변경
```python
catchup = int(settings.get("schedule", {}).get("departure_catchup_minutes", 45))
expiry = (
    alert.plan.departure_time if alert.alert_type == "prep"
    else alert.plan.departure_time + timedelta(minutes=catchup)
)
```

### settings.yaml 추가
```yaml
schedule:
  alert_window_minutes: 15
  departure_catchup_minutes: 45   # 신규
```
