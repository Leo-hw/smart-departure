# T-010 — 견고성 + 놓친 알림 catch-up

**상태**: 🟡 READY  
**담당**: Codex  
**의존**: T-007

---

## 배경

2026-04-29 GitHub Actions 실행이 Google Calendar 호출 중 SSL 핸드셰이크 타임아웃으로 크래시(`startup failed: _ssl.c:999 ... timed out`)하여 알림이 전혀 발송되지 않음.

원인 두 겹:
1. **견고성 부족** — Calendar 호출에 timeout/retry 없음 (Maps는 fallback 하는데 Calendar만 무방비). 일시적 네트워크 장애 한 번에 전체 크래시.
2. **설계 취약점** — 알림 윈도우 ±15분 + cron 30분 간격인데, 한 번 실패/지연하면 catch-up이 없어 알림이 영구 유실.

추가로 GitHub Actions는 `.runtime/`가 매 실행마다 휘발되어 **dedup이 실행 간에 동작하지 않는 상태**임.

---

## 목표

1. Calendar 호출 재시도/타임아웃으로 일시적 장애에 견디게 한다.
2. `.runtime/` 상태를 GitHub Actions cache로 실행 간 보존한다.
3. 정시 윈도우를 놓친 알림을 만료 전까지 늦게라도 발송(catch-up)한다.
4. 실행 크래시 시 조용히 죽지 않고 Discord로 통보한다.

---

## Acceptance Criteria

### A. Calendar 견고성
- [ ] `core/calendar_service.py`의 `events().list().execute()`에 `num_retries=3` 적용
- [ ] Calendar 빌드/호출에 timeout 적용 (httplib2 `Http(timeout=...)` 또는 동등 수단, 기본 10초)
- [ ] 재시도 모두 실패 시에도 `main()`이 크래시하지 않고 D(에러 알림) 경로로 처리

### B. 상태 영속화 (GitHub Actions)
- [ ] `.github/workflows/departure_check.yml`에 `actions/cache`로 `.runtime/` 보존
- [ ] 롤링 키: `key: runtime-${{ github.run_id }}`, `restore-keys: |\n  runtime-`
- [ ] dedup 만료를 `event_start + 60분`으로 변경 (기존 단순 TTL 60분 대체)
- [ ] dedup 레코드에 `event_start`를 저장해 만료 판정에 사용

### C. catch-up 발송
- [ ] 각 알림에 정시 윈도우(`target ± alert_window_minutes`)와 catch-up 만료 시각 부여
  - 준비 단계 만료: 해당 일정의 `departure_time`
  - 출발 만료: `departure_time + 30분`
- [ ] `now`가 정시 윈도우 안 → 정상 발송
- [ ] `now`가 정시 윈도우 밖 ~ 만료 이내 → catch-up 발송 + **늦음 표기**
- [ ] `now`가 만료 초과 → 폐기(발송 안 함)
- [ ] 늦음 표기 포맷: `⚠️ 늦었어요 — 원래 {target} 예정, 지금 {now}`
- [ ] 준비 단계 여러 개가 과거-미발송이면 `target ≤ now`인 것 중 **가장 최근 하나만** 발송
- [ ] 봉인된(더 이른) 준비 단계는 dedup에 `status="skipped"`로 기록해 재발송 방지
- [ ] 이미 발송 성공한 알림은 catch-up 대상에서 제외 (dedup 존중)

### D. 실패 시 에러 알림
- [ ] `main()`이 예외로 종료될 때 Discord로 `[smart-departure] 실행 실패: {exc}` 전송 시도
- [ ] 에러 알림 전송 자체가 실패해도 stderr 기록 후 exit 1 (무한 루프 방지)
- [ ] 에러 알림은 dedup 대상 아님 (매 실패마다 통보)

### E. 테스트
- [ ] catch-up 윈도우/만료 판정 단위 테스트 (정시/늦음/만료초과 3케이스)
- [ ] 준비 단계 다중 누락 시 "가장 최근만" 선택 + 나머지 skipped 기록 테스트
- [ ] dedup `event_start + 60분` 만료 테스트
- [ ] Calendar 재시도 동작 테스트 (mock으로 1~2회 실패 후 성공)

## 구현 상세

### catch-up 판정 의사코드
```python
def classify_alert(alert, now, settings):
    window = settings["schedule"]["alert_window_minutes"]
    on_time_lo = alert.alert_time - window
    on_time_hi = alert.alert_time + window
    expiry = (
        alert.departure_time if alert.alert_type == "prep"
        else alert.departure_time + timedelta(minutes=30)
    )

    if now < on_time_lo:
        return "pending"      # 아직
    if on_time_lo <= now <= on_time_hi:
        return "on_time"      # 정상 발송
    if on_time_hi < now <= expiry:
        return "catch_up"     # 늦게라도 발송 (늦음 표기)
    return "expired"          # 폐기
```

### 준비 단계 "가장 최근만" 선택
```python
# 같은 event_id의 prep 단계 중 on_time/catch_up 상태이고 target<=now 인 것들
due_prep = [a for a in prep_alerts if a.target <= now and status in {"on_time","catch_up"}]
if due_prep:
    latest = max(due_prep, key=lambda a: a.target)
    send(latest)
    for a in due_prep:
        if a is not latest:
            dedup.record(a, status="skipped")   # 봉인
```

### dedup 레코드 구조 (변경)
```json
{
  "{event_id}_{alert_key}": {
    "status": "sent" | "skipped",
    "sent_at": "2026-04-30T14:50:00+09:00",
    "event_start": "2026-04-30T16:00:00+09:00"
  }
}
```
- 만료: `now > event_start + 60분`인 레코드는 정리

### GitHub Actions cache 추가
```yaml
      - name: Restore runtime state
        uses: actions/cache@v4
        with:
          path: .runtime
          key: runtime-${{ github.run_id }}
          restore-keys: |
            runtime-
```
> 주의: actions/cache는 키 단위 불변. 롤링 키로 매 실행 최신 복원 후 새 캐시 저장. 캐시 전파에 수 분 지연이 있을 수 있으나 30분 cron 간격에서는 문제없음.
